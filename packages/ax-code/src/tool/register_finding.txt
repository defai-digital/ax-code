Register a single review/debug/qa finding as a structured artifact. Call this ONLY when you have already been asked to perform a code review, debug analysis, or QA check — never call it from general coding tasks. Each call produces ONE finding; call it as many times as you have findings.

When you have findings to surface, prefer this tool over writing them in prose. Prose findings cannot be exported, dedup'd across runs, or routed into review tooling. Tool-emitted findings can.

Input (all required unless noted):
- workflow: "review" | "debug" | "qa" — which assurance lane produced the finding
- category: "bug" | "security" | "regression_risk" | "behavior_change" | "missing_verification" | "migration_safety"
- severity: "CRITICAL" | "HIGH" | "MEDIUM" | "LOW" | "INFO" (upper-case)
- summary: one-line headline, max 200 characters, no trailing period
- file: repo-relative POSIX path of the affected file
- anchor: { kind: "line", line: <number>, endLine?: <number> } OR { kind: "symbol", symbolId: <CodeNodeID> }
- rationale: short explanation of why this is a finding (1–3 sentences). Not the same as evidence.
- evidence: array of strings — file:line refs, quoted snippets, command output. Pass [] only if severity is LOW or INFO.
- suggestedNextAction: a concrete next step the user can take. Not a patch.
- confidence: optional number in [0, 1]. Emit when you have a meaningful confidence signal (heuristic-derived findings); omit for deterministic findings.
- ruleId: optional string of form "<axcode|policy|user>:<kebab-name>", e.g. "axcode:bug-empty-catch".
- evidenceRefs: optional array of typed back-references. Use { kind: "verification", id: "<16-char hex>" } to cite a VerificationEnvelope from this session (e.g. the typecheck/lint/test run that produced the evidence). The id is computed deterministically from envelope content via computeEnvelopeId — do not invent ids; only cite envelopes you have observed in the session's tool results. Verification refs must belong to the same workflow as this finding.
- tool: optional identifier of the producing tool/workflow (e.g. "review", "audit"). Defaults to "review".

Output:
- findingId: deterministic 16-char lowercase hex hash of (workflow + category + file + anchor + ruleId). Repeat callers for the same defect get the same id, enabling dedup across runs.
- A one-line confirmation suitable for chat output.

Hard rules:
- Do NOT also write findings as prose blocks in your reply. Pick this tool OR prose, not both.
- One finding per call. If you find five issues, call this tool five times.
- Do not invent a `file` that does not exist — anchor every finding to a real path you have read or one that appears in the diff.
- Do not invent symbol ids — use a `line` anchor unless you have already resolved the symbol via findSymbol or the code intelligence tool.
- If you are uncertain whether something is a finding, do not call this tool. Bring it up in prose for human judgment instead.

When NOT to use:
- General coding edits — use edit/write/apply_patch.
- Asking the user a question — use the question tool / prose.
- Recording your own thoughts — keep them in prose; this tool is for actionable, file-anchored findings only.
