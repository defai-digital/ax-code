Help with the native AX Wiki for this project (path: ${path}).

AX Wiki is the source-backed semantic layer for architecture, module intent, and workflows. It complements `code_intelligence` and LSP, which remain authoritative for precise symbols and relationships. Guide: docs/integrations/wiki.md

User request: $ARGUMENTS

Parse $ARGUMENTS as an optional subcommand (case-insensitive):

| Arg | Action |
|-----|--------|
| empty / help | Summarize `ax-code wiki doctor` guidance and recommend next steps |
| status | Run or describe `ax-code wiki status` |
| doctor | Run or describe `ax-code wiki doctor` |
| plan | Preview `ax-code wiki plan` without model calls |
| lint | Run `ax-code wiki lint` |
| cards | Run `ax-code wiki cards` |
| related \<Symbol\> | Run `ax-code wiki related <Symbol>` |
| ensure / ensure-agents | Run `ax-code wiki ensure-agents` |
| generate / init | Run `ax-code wiki generate` |
| update / refresh | Run `ax-code wiki update` |
| otherwise | Read `ax-wiki/quickstart.md` or `.ax-code/wiki-cards.md`, drill into relevant pages, then answer |

Rules:

1. Prefer the real CLI (`ax-code wiki …`) over duplicating compiler behavior inside a session.
2. Generation uses AX Code's configured providers and model; no separate wiki runtime is required.
3. After generate/update, keep AGENTS pointers current with `ax-code wiki ensure-agents`.
4. Keep AGENTS.md thin; never paste the whole wiki into it.
5. Preserve maintainer-owned text inside `AX-WIKI:PROTECTED` markers.
6. For architecture questions, read the wiki index and drill down; for callers/callees use `code_intelligence`.
