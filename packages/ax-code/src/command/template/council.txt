Run a multi-provider council review (advisory). Use the **council** tool **as the primary action**.

Question / focus: $ARGUMENTS

## HARD RULES (do not skip)

1. Call the **council** tool within the first 1–2 tool rounds. Do **not** call `task_parallel` or multi-explore digs first.
2. Put brief repo facts into the council `context` argument (stack, 3–10 key paths). Optional: one `read`/`list`/`bash` only — then **council**.
3. Never run four “very thorough” explores before council. Multi-provider models need a short brief, not a full monorepo map.
4. If council reports insufficient providers, stop and tell the user to connect ≥2 providers. Do not invent consensus.
5. After council returns, summarize **consensus / strict majority / minority / singleton** for the user. Do not apply patches solely because models agreed.

## Task fit

- **council** (this flow): quality findings, security, architecture trade-offs, “good vs worse” ratings.
- **arena plan**: compare implementation *approaches* only.
- **arena implement**: competing patches in git worktrees (not for pure review).

## Call shape

- `kind: review` for code quality; `kind: design` for architecture trade-offs.
- Optional `debateRounds: 1` only if stakes are high and providers are already connected (maximum 3; stops early on convergence).
- Config is re-read each call (`modes.council.enabled` defaults on).
