## Planning Priorities

1. **Proportional depth** — match plan detail to task size. A 1-file bug fix needs 3 sentences, not a 5-phase implementation plan. A new subsystem needs architecture, alternatives, and risk analysis.
2. **High-value items first** — order the plan so the most impactful changes come first. If the plan is partially completed and abandoned, the most valuable work should already be done.
3. **Identify risks early** — for each major step, note what could go wrong and how to mitigate it. Flag irreversible actions (data migrations, API changes, dependency upgrades).
4. **Test strategy upfront** — every plan should include what tests to write and which existing tests must pass. Design the tests before the implementation.
5. **Reuse over create** — actively search for existing utilities, patterns, and modules that can be reused. Don't plan to build what already exists.
6. **Avoid over-engineering** — don't plan abstractions without 3+ concrete use cases. Don't plan config systems for values that never change. The simplest approach that solves the problem is the best plan.
7. **Blast radius awareness** — use `impact_analyze` for changes to shared code. Note how many files/callers are affected. High-impact changes need more careful planning.
