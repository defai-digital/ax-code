## Purpose

This document turns the phase plan in the ratatui migration PRD into an execution backlog.

It is a companion to:

- `automatosx/adr/ADR-001-ratatui-bundled-renderer-and-opentui-rollback-only.md`
- `automatosx/prd/v4.1.0-ratatui-migration.md`

Execution policy:

- no production code changes should start until this backlog and the companion PRD are reviewed and accepted
- `v4.1.0` release gates must stay separate from the longer-term renderer replacement contract
- `tui.*` and `tui.control.*` are excluded from the Rust client's release contract

## Ownership Model

Use code-slice ownership instead of informal shared ownership.

| Owner slice | Primary responsibility | Primary files | Review responsibility |
|---|---|---|---|
| Docs and scope owner | ADR/PRD scope, support matrix, release-vs-long-term gate split | `automatosx/adr/*.md`, `automatosx/prd/*.md`, `packages/ax-code/src/cli/cmd/tui/renderer-contract.ts`, `packages/ax-code/src/cli/cmd/tui/performance-criteria.ts` | reviews every phase gate before implementation starts |
| Server contract owner | renderer-neutral routes, OpenAPI, operation allowlist, fixture truth | `packages/ax-code/src/server/routes/*.ts`, `packages/sdk/openapi.json` | reviews Phases 1, 3, and 4 contract changes |
| Native TUI owner | ratatui runtime, terminal lifecycle, event loop, UI state, Rust integration tests | `crates/ax-code-tui/**`, `crates/Cargo.toml`, `crates/.cargo/**` | reviews Phases 2-6 native runtime work |
| CLI integration owner | CLI selection, attach/spawn plumbing, diagnostics bridge, release-path messaging | `packages/ax-code/src/cli/cmd/tui/*.ts`, `packages/ax-code/src/cli/cmd/doctor*.ts`, `script/setup-cli.ts`, `packages/ax-code/script/build.ts` | reviews Phases 2 and 5 integration work |
| Quality owner | fixtures, parity oracles, PTY smoke, release checks, failure-path cleanup tests | `packages/ax-code/test/cli/tui/**`, `packages/ax-code/test/cli/*.test.ts`, `crates/ax-code-tui/tests/**` | owns go/no-go evidence for every phase |

## Cross-Phase Rules

- every phase must end with written go/no-go evidence, not only implementation progress
- tests that define a contract should be written or updated before implementation under that contract starts
- when a phase reveals scope pressure, update ADR/PRD/backlog first; do not silently expand `v4.1.0`
- deferred features remain on the long-term replacement track until explicitly promoted

## Phase Backlog

## Phase Dependencies

| Phase | Depends on | Why |
|---|---|---|
| 0 | none | decision freeze comes first |
| 1 | 0 | contract work cannot start before scope and gate split are approved |
| 2 | 1 | runtime should consume an approved allowlist and fixture plan |
| 3 | 2 | read-only UI depends on launchable runtime and terminal guard |
| 4 | 3 | write-path work should not start before read-only session flow is stable |
| 5 | 4 | default flip and release hardening require end-to-end usable chat flow |
| 6 | 5 | migration close should happen only after release-path stability is proven |

## Minimum Evidence Per Phase

| Phase | Required evidence before merge |
|---|---|
| 0 | approved ADR/PRD/backlog diff and explicit `v4.1.0` gate split |
| 1 | approved operation allowlist, fixture inventory, and projection-route contract tests |
| 2 | native launch and attach smoke, terminal cleanup proof, and startup diagnostics proof |
| 3 | fixture-backed snapshots, reconnect parity evidence, and read-only session walkthrough |
| 4 | prompt/abort/dialog integration evidence and focus-ownership proof |
| 5 | packaged runtime proof, PTY smoke results, doctor/setup-cli proof, and release checklist |
| 6 | removal diff, full release-path proof, and long-term contract clearance |

## Initial Recommended Sequencing

1. finish Phase 0 docs review and approve the gate split
2. review the Phase 1 allowlist shape before any route or fixture implementation starts
3. treat Phase 2 terminal lifecycle and runtime skeleton as the first implementation milestone
4. do not start Phase 5 default-flip work until Phase 4 interactive flows are already usable
