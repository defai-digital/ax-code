# PRD: AX Code v4.1.0 - ratatui TUI Migration

**Date:** 2026-04-23
**Status:** Draft
**Scope:** Internal
**Last reviewed:** 2026-04-23
**Owner:** ax-code agent

---

## Problem Statement

AX Code's current TUI depends on OpenTUI (`@opentui/core`, `@opentui/solid`) plus a Bun Worker bridge. That stack has already forced product-level workarounds because bundled Bun binaries can hang in TUI startup and rendering paths. Source-launcher mitigation is not a durable release strategy for bundled, offline, or distribution builds.

The migration target is to move the shipped renderer path onto Rust (`ratatui` + `crossterm`) while preserving the existing Bun server, HTTP API, SSE model, and backend logic.

## Decision Reference

Architecture decisions for this migration are fixed in [ADR-001](../adr/ADR-001-ratatui-bundled-renderer-and-opentui-rollback-only.md).

This PRD is intentionally phase-oriented:

- ADR owns the decision and tradeoffs
- PRD owns sequencing, scope, gates, and execution checkpoints

## Goals

- Ship `v4.1.0` with `ratatui` as the only bundled/release renderer
- Remove OpenTUI from the bundled runtime path without rewriting backend logic into Rust
- Keep `v4.1.0` scope tight enough to be shippable without promising full parity
- Preserve diagnosability, reconnect behavior, auth/header behavior, and packaging determinism
- Use explicit contracts, fixtures, and tests instead of line-count or checklist optimism

## Non-Goals

- `v4.1.0` does not remove `vendor/opentui/` or OpenTUI dependencies yet
- `v4.1.0` does not deliver full parity across every dialog, sidebar surface, or advanced transcript feature
- `v4.1.0` does not rewrite the server, session logic, or SDK into Rust
- `v4.1.0` does not keep OpenTUI as a bundled runtime fallback

## Support Matrix

| Environment | Supported renderer path |
|---|---|
| bundled / release | `ratatui` only |
| local source / dev | `ratatui` primary; OpenTUI may remain temporarily for debugging only |
| rollback | version rollback or source/dev troubleshooting |

Rules:

- no automatic fallback from ratatui to OpenTUI
- requesting OpenTUI in bundled/release mode must fail fast
- release rollback is done by product version rollback, not by in-process renderer switching
- during Phases 2-4, local/dev ratatui may exist as an explicit opt-in path, but it is not considered the primary local/dev path until Phase 5 flips bare `ax-code`, `setup-cli`, and `doctor` messaging together

## Best-Practice Guardrails

These guardrails are mandatory across all phases:

- one crossterm event model only:
  use `EventStream` or `read/poll`, never both
- one state writer only:
  input, timers, and network updates must become actions into a reducer
- one render ownership model only:
  widgets render state; they do not own async side effects
- no network or filesystem I/O inside render code
- terminal capability setup and cleanup must live in a guard module with explicit error-path coverage
- graceful shutdown must use coordinated task cancellation and task joining
- additive server projection routes are allowed only for read models, never for duplicating business logic
- operation allowlists must be explicit and renderer-neutral; `tui.*` and `tui.control.*` are not valid release contracts for the Rust client
- bundled support matrix must stay simple and honest:
  ratatui-only release path, version rollback instead of hidden renderer fallback
- release gates must be versioned to the announced scope; a broader long-term replacement contract may not silently block a narrower `v4.1.0` release
- fixture-backed snapshot tests are required for pure UI projections
- PTY smoke tests are required for startup, resize, paste, interrupt, reconnect, and cleanup paths

## Anti-Patterns To Avoid

- silently falling back from ratatui to OpenTUI in bundled/release mode
- rebuilding large amounts of current bootstrap orchestration in Rust from scattered low-level routes when a stable read-model contract would be safer
- using renderer-specific control routes such as `tui.*` or `tui.control.*` as the Rust client's primary contract
- mixing terminal input APIs in one process
- letting components/widgets own HTTP clients or spawn tasks directly
- relying on "first frame painted" as proof that the renderer is release-ready
- keeping a broader long-term replacement contract as an implicit blocker for a narrower `v4.1.0` scope without first splitting or versioning it
- deferring packaging verification until the end of the implementation
- declaring parity based on line counts or visual inspection instead of contracts, fixtures, and tests

## Contract Layers

This migration has two different contract layers that must not be conflated:

- long-term renderer replacement contract:
  `packages/ax-code/src/cli/cmd/tui/renderer-contract.ts` and `packages/ax-code/src/cli/cmd/tui/performance-criteria.ts` remain useful as broader replacement references and may include post-`v4.1.0` expectations
- `v4.1.0` release gate:
  only the must-have scope in this PRD is allowed to block release sign-off for `v4.1.0`

Phase 0 must split or version any inherited gate that still covers explicitly deferred scope. No deferred item may remain a `v4.1.0` release blocker only because it was required by an older full-replacement contract.

## Phase Overview

| Phase | Focus | Outcome |
|---|---|---|
| 0 | ADR and gate freeze | fixed decision boundary and support matrix |
| 1 | Contract and fixture extraction | reviewable bootstrap/session contracts and test fixtures |
| 2 | Runtime foundation | ratatui crate, lifecycle, terminal guard, renderer selection wiring |
| 3 | Read-only session flow | usable navigation and live transcript consumption |
| 4 | Interactive core | prompt submit, abort, permission/question flows |
| 5 | Release hardening | packaged ratatui-only release path with diagnostics and tests |
| 6 | Migration close | remove temporary OpenTUI escape hatches after stability proof |

## Phase Plan

### Phase 0: ADR, Gates, and Scope Freeze

Objective:

- freeze the decision boundary before implementation starts

In scope:

- ADR for renderer strategy
- support matrix for bundled, dev, and rollback behavior
- go/no-go gates for `v4.1.0`
- narrow `v4.1.0` feature scope

Deliverables:

- `automatosx/adr/ADR-001-*.md`
- this PRD updated to phase form
- explicit unsupported behavior for OpenTUI in bundled/release mode
- split or version the current renderer replacement contract into:
  a `v4.1.0` release gate and a broader post-`v4.1.0` replacement track

Exit criteria:

- `ADR-001` is accepted and no longer reopens the renderer strategy debate for implementation PRs
- team can answer "what is supported in `v4.1.0`?" in one paragraph
- no document still claims OpenTUI is a bundled runtime fallback
- no explicitly deferred item still blocks `v4.1.0` through an inherited legacy gate
- guardrails and anti-patterns are explicit enough that implementation reviews can reference them directly

### Phase 1: Server Contracts and Oracle Fixtures

Objective:

- give the Rust client a small, reviewable, testable contract surface

In scope:

- decide whether additive bootstrap/session snapshot routes are needed
- if needed, add renderer-neutral projection routes backed by current sources of truth
- capture canonical fixtures for bootstrap, session hydration, permission, question, and transcript state
- map existing TS tests that become parity oracles

Deliverables:

- explicit renderer-neutral operation allowlist based on server `operationId`s, with `tui.*` and `tui.control.*` excluded from the ratatui release contract
- fixture set for Rust snapshot/integration tests
- contract tests for any new projection routes

Exit criteria:

- Rust client does not need to reverse-engineer scattered bootstrap behavior
- parity is grounded in fixtures and tests, not prose
- any additive contract is clearly a read model, not duplicated business logic
- the allowlist is concrete enough that a reviewer can reject `tui.*` coupling on sight

Not in scope:

- rich transcript styling
- advanced dialog parity

### Phase 2: Runtime Foundation

Objective:

- make the ratatui process launch, connect, render, and shut down correctly

In scope:

- `crates/ax-code-tui` workspace member
- CLI plumbing for local spawn and attach modes

- terminal guard for raw mode, alternate screen, mouse, focus, and bracketed paste

Deliverables:

- `crates/ax-code-tui/src/runtime.rs`
- `crates/ax-code-tui/src/app.rs`

- `crates/ax-code-tui/src/diagnostics.rs`

Exit criteria:

- `ax-code --renderer ratatui` launches and paints a non-empty frame
- `ax-code attach <url> --renderer ratatui` connects
- terminal cleanup works on success and failure
- runtime structure already enforces the single event/action/reducer/render model

Not in scope:

- session creation workflow
- prompt submission

### Phase 5: Release Hardening

Objective:

- ship `v4.1.0` as a real ratatui-only bundled/release path

In scope:

Exit criteria:

- migration window can close without losing critical supportability
- repository no longer carries two renderer stories

## Test and Gate Strategy

### Contract and fixture gates

- every Rust operation must map to an explicitly allowlisted existing server `operationId`
- `tui.*` and `tui.control.*` are excluded from the ratatui release contract
- any additive projection route must be versioned in OpenAPI and covered by contract tests
- fixture set must cover bootstrap, session hydration, permission, question, and transcript state
- fixture names and ownership should map to product concepts, not ad hoc renderer internals

### Scope-alignment gates

- `renderer-contract.ts` and `performance-criteria.ts` are long-term replacement references, not automatic `v4.1.0` release blockers
- any inherited gate that covers an explicitly deferred item must be split, versioned, or moved to the post-`v4.1.0` track before implementation PRs rely on it
- no deferred feature may remain a release blocker only because a legacy test or checklist still encodes broader replacement scope

### Runtime and terminal gates

- one crossterm input model only:
  `EventStream` or `read/poll`, never both
- raw mode and terminal capture cleanup must be tested on error paths
- graceful shutdown must cancel child tasks and wait for completion
- render code must remain side-effect free

### Renderer and release gates

- startup diagnostics emitted
- reconnect behavior matches current product expectations
- bundled/release build works without OpenTUI
- release notes explain narrow scope and rollback path
- fail-fast unsupported behavior for OpenTUI in bundled/release mode is covered by tests

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Scope expands back to full parity | High | High | keep `v4.1.0` must-have scope narrow and phase-gated |
| Rust client duplicates fragile server orchestration | High | High | add projection contracts where duplication would be brittle |
| Bundled packaging fails late | High | High | treat packaging as a Phase 2-5 concern, not a final step |
| Reconnect/bootstrap behavior regresses | Medium | High | preserve current semantics through fixtures and integration tests |
| Terminal cleanup is broken on failure paths | Medium | High | isolate terminal lifecycle in a guard and test cleanup explicitly |
| No bundled fallback increases pressure on release quality | High | Medium | use version rollback and scope discipline instead of runtime fallback |
| Temporary OpenTUI escape hatches linger too long | Medium | Medium | time-box migration close and keep removal criteria explicit |

## Success Criteria

`v4.1.0` is successful if:

- bundled/release builds use ratatui only
- requesting OpenTUI in bundled/release mode fails fast and clearly
- users can complete a normal chat session in ratatui with prompt submit, interrupt, and permission/question handling
- reconnect, auth, directory headers, and diagnostics behave correctly
- release notes describe the narrower scope and rollback path honestly

The migration is complete only when:

- ratatui clears the long-term renderer replacement contract and performance gates
- packaged releases remain deterministic across supported targets
- temporary OpenTUI escape hatches can be removed without losing supportability
