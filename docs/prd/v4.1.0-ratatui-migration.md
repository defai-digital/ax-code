# PRD: AX Code v4.1.0 — ratatui TUI Migration

## Context

OpenTUI (Zig + bun:ffi) causes TUI hangs in compiled Bun binaries due to upstream Bun bugs
(oven-sh/bun#26762, #29124, #27766). The same issue affects OpenCode and Claude Code.
v4.0.13 works around it with a source launcher, but the root cause — dependency on Bun's
compiled binary runtime for TUI rendering — remains.

v4.1.0 replaces OpenTUI with ratatui (pure Rust), eliminating the Bun dependency from the
rendering pipeline entirely.

## Architecture

```
┌─────────────────────┐      HTTP + SSE       ┌─────────────────────┐
│   ax-code-tui       │  ◄─────────────────►  │   ax-code server    │
│   (Rust / ratatui)  │   localhost:4096       │   (Bun / TypeScript)│
│   crates/ax-code-tui│                        │   packages/ax-code/ │
└─────────────────────┘                        └─────────────────────┘
```

- **ax-code-tui**: standalone Rust binary, compiled with `cargo build`
- **ax-code server**: existing Hono HTTP server, runs via `bun run` (source launcher)
- **Communication**: HTTP REST API + SSE event stream over localhost
- **Launch**: `ax-code` command starts server, then spawns ratatui binary, passes port

## Scope — v4.1.0 MVP (full feature parity)

### Must have
- Prompt input (multiline, paste, history, autocomplete)
- Message stream rendering (text, reasoning, tool calls, tool results)
- Markdown rendering with syntax highlighting
- Session list, create, switch, delete
- Model / provider picker, agent selector
- Permission dialogs (allow/deny/always allow)
- Question dialogs
- Status bar (model, tokens, step count, session status)
- Keybinding system (Ctrl+C interrupt, Ctrl+K commands, vim-style nav)
- Sidebar (session tree, task list)
- Toast notifications
- `/` command palette

### Won't change
- Server HTTP API (all routes stay the same)
- SSE event system
- SDK (packages/sdk/js/)
- VS Code extension
- All backend logic (agents, tools, providers, sessions, permissions)

## Code changes

### Remove
- `packages/ax-code/src/cli/cmd/tui/` — 163 files, ~8,800 lines of TSX/TS (OpenTUI components)
- `vendor/opentui/` — vendored OpenTUI
- `@opentui/core`, `@opentui/solid` dependencies
- `packages/ax-code/src/cli/cmd/tui/worker.ts` — Bun Worker thread (no longer needed)

### Add
- `crates/ax-code-tui/` — new Rust crate in the existing workspace
  - `Cargo.toml` — deps: ratatui, crossterm, tokio, reqwest, eventsource-client, serde, serde_json, syntect, clap
  - `src/main.rs` — entry point, arg parsing, server connection
  - `src/app.rs` — app state machine (loading → home → session → dialog)
  - `src/client.rs` — HTTP client wrapper (reqwest) for ax-code server API
  - `src/sse.rs` — SSE event stream consumer (real-time sync)
  - `src/ui/` — ratatui widget modules:
    - `prompt.rs` — text input with history and autocomplete
    - `messages.rs` — message list rendering (text, tool calls, reasoning)
    - `markdown.rs` — markdown → ratatui spans with syntax highlighting (syntect)
    - `sessions.rs` — session list / home screen
    - `status.rs` — bottom status bar
    - `dialog.rs` — permission / question / model picker dialogs
    - `sidebar.rs` — session tree, task list
    - `toast.rs` — notification overlay
    - `command.rs` — `/` command palette
    - `theme.rs` — color theme (matching current ax-code palette)
  - `src/keybind.rs` — keybinding registry
  - `src/event.rs` — crossterm event loop + SSE event merging

### Modify
- `packages/ax-code/src/cli/cmd/tui/` → replace with thin launcher that starts server + spawns Rust binary
- `script/setup-cli.ts` — build step calls `cargo build --release` for ax-code-tui
- `crates/Cargo.toml` — add `ax-code-tui` to workspace members
- `package.json` — add `build:tui` script

## Server API surface (ratatui client must implement)

The existing HTTP API is complete — the current TUI already uses it via the SDK.
Key endpoints the ratatui client needs:

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/session` | List sessions |
| POST | `/session` | Create session |
| DELETE | `/session/:id` | Delete session |
| GET | `/session/:id/message` | Get messages |
| POST | `/session/:id/prompt_async` | Send prompt |
| POST | `/session/:id/abort` | Cancel generation |
| GET | `/event` | SSE event stream (all real-time updates) |
| GET | `/provider` | List providers + models |
| GET | `/agent` | List agents |
| GET | `/config` | Get config |
| POST | `/permission/:id/resolve` | Resolve permission request |
| POST | `/question/:id/resolve` | Resolve question request |
| POST | `/session/:id/command_async` | Run `/` command |

## Implementation phases

### Phase 1: Skeleton + server connection (2 days)
- Create `crates/ax-code-tui/` with Cargo.toml
- Implement `main.rs` — parse args (--port, --password), connect to server
- Implement `client.rs` — reqwest wrapper with basic auth
- Implement `sse.rs` — connect to `/event` SSE stream
- Implement minimal `app.rs` — render "Connected to ax-code server" on screen
- Verify: `cargo run -- --port 4096` connects and receives events

### Phase 2: Home screen + session management (2 days)
- Implement `sessions.rs` — list sessions from GET `/session`
- Create / delete / switch sessions
- Navigation: arrow keys, enter, delete
- Implement `status.rs` — bottom bar with version, model name

### Phase 3: Core chat loop (3 days)
- Implement `prompt.rs` — multiline text input with history
- Implement `messages.rs` — render text, tool calls, reasoning parts from SSE events
- Wire prompt submit → POST `/session/:id/prompt_async`
- Wire Ctrl+C → POST `/session/:id/abort`
- Implement streaming display (text deltas arrive via SSE)

### Phase 4: Markdown + syntax highlighting (2 days)
- Implement `markdown.rs` — parse markdown to ratatui styled spans

### Phase 6: Advanced UI (2 days)

### Phase 7: Integration + cleanup (2 days)

## Risks

| Risk | Mitigation |
|------|-----------|
| SSE reconnection reliability | Use `eventsource-client` with auto-reconnect; match resilient-stream.ts logic |
| Markdown rendering quality | Start with basic, iterate; syntect handles code highlighting |
| Crossterm terminal compatibility | Widely tested across terminals; same matrix as current |
| Build time for Rust crate | Incremental builds; only full rebuild on CI |
| Two-process lifecycle management | Server watches child exit; TUI watches parent pid |
