## 1. Programmatic SDK (`@ax-code/sdk/programmatic`)

### What It Is
A direct agent instantiation SDK that lets developers use ax-code as a library. Developers write `agent.run("fix the bug")` and the agent autonomously reads files, runs commands, edits code, and returns the result.

### API
```typescript
import { createAgent } from "@ax-code/sdk/programmatic"

const agent = await createAgent({ directory: ".", auth: { provider: "xai", apiKey: "key" } })
const result = await agent.run("fix the login bug")
const text = await agent.stream("explain this code").text()
const session = await agent.session()  // multi-turn
const models = await agent.models()    // 78 models
const tools = await agent.tools()      // 15 tools
await agent.dispose()
```

### Production Features
- Typed errors: `ProviderError`, `TimeoutError`, `ToolError`, `DisposedError`, `AgentNotFoundError`
- Stream convenience: `.text()`, `.result()`, `.on("text", cb)`, `.done()`
- Auto-retry on transient failures (429, 500) with exponential backoff
- Timeout support on `createAgent()` and `agent.run()`
- Direct API key auth — no local setup needed
- Auto env var detection (XAI_API_KEY, GEMINI_API_KEY)
- Model/tool discovery methods

### Comparison: ax-code SDK vs OpenCode HTTP SDK

#### Developer Experience
| Task | OpenCode HTTP SDK | ax-code Programmatic SDK |
|---|---|---|
| Setup | 15+ lines (spawn server, create client, subscribe events, handle SSE) | 2 lines (`createAgent` + `run`) |
| Send prompt | `sdk.session.create()` → `sdk.event.subscribe()` → `sdk.session.prompt()` → `for await` loop | `agent.run("prompt")` |
| Stream text | Manual SSE parsing, event type checking | `agent.stream("prompt").text()` |
| Multi-turn | Manual session ID tracking | `agent.session()` → `session.run()` |
| Error handling | Generic HTTP errors, parse response body | Typed `ProviderError`, `TimeoutError`, etc. with `.code`, `.status` |
| Tool discovery | Separate API call | `agent.tools()` |
| Model discovery | Separate API call | `agent.models()` |

#### Reliability
| Issue | OpenCode HTTP SDK | ax-code Programmatic SDK |
|---|---|---|
| Event subscription race condition | Yes — if you subscribe after prompt, you miss events | No — events handled internally |
| Server process management | Must spawn, health check, teardown | None needed — runs in-process |
| Error recovery | Manual retry logic | Built-in `maxRetries` with backoff |
| Timeout handling | No built-in support | Built-in `TimeoutError` |

#### Performance — Real Benchmark (same machine, same prompt, same provider)

**Test conditions:** Both tests ran on the same machine, same directory, same prompt ("What is 2+2? One word."), same LLM provider (Z.AI/GLM), within minutes of each other.

**External Developer Perspective (this is how someone using the SDK would experience it):**

| Step | OpenCode (External Server Spawn) | ax-code (Programmatic SDK) |
|---|---|---|
| Step 1: Setup | Spawn server process: **3,158ms** | createAgent(): **744ms** |
| Step 2: Create session | HTTP session.create(): **15,969ms** | Included in run: **0ms** |
| Step 3: Subscribe events | event.subscribe(): **0ms** | Handled internally: **0ms** |
| Step 4: Prompt + response | **STUCK** (SSE race condition bug) | agent.run(): **5,470ms** |
| **TOTAL** | **19,127ms+ (never completed)** | **8,190ms (completed, result: "4")** |

**Key findings:**
1. **Startup: 3,158ms vs 744ms** — ax-code SDK is **4.2x faster** (no server process spawn)
2. **OpenCode got stuck** — SSE event subscription race condition, never returned a result
3. **ax-code completed successfully** in 8.2 seconds with correct result
4. **OpenCode's session.create took 15,969ms** — includes provider initialization on first HTTP call

**In-Process Comparison (both running inside the same process):**

| Metric | OpenCode HTTP SDK (in-process) | ax-code Programmatic SDK | Notes |
|---|---|---|---|
| Bootstrap/init | ~500-800ms | ~500-800ms | Same — both call bootstrap() |
| Session + subscribe | ~47ms | ~0ms (included in run) | SDK handles internally |
| Per-call LLM time | 3-13s | 3-13s | Same — depends on provider |
| Total first call | ~4-14s | ~4-14s | Same speed |

**Speed summary:** In-process performance is identical — both use the same bootstrap() and Server.Default().fetch(). The speed advantage is ONLY for external developers who skip the server process spawn (saves 1.5-3s). Our real advantage over OpenCode is developer experience (2 lines vs 15+), reliability (no SSE race condition), and capabilities (typed errors, stream helpers, discovery, hooks).
