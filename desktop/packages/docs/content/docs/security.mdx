# Security

AX Code Desktop gives broad access to your machine and source code. It is therefore restricted to loopback addresses and is intended for office and offline workstations.

Remote hosts, SSH instances, LAN binding, reverse proxies, mDNS discovery, and Cloudflare or other public tunnels are disabled. Legacy settings for these features are ignored and removed during startup.

## Set a UI password

Start AX Code Desktop with a password and the browser UI asks for it:

```bash
ax-code-desktop --ui-password be-creative-here
```

You can also set it with the `AX_CODE_DESKTOP_UI_PASSWORD` environment variable instead of putting it on the command line. After signing in, AX Code Desktop remembers the device for a while so you're not asked every time.

A password remains useful on a shared workstation, but it does not enable network access.

## Passkeys

Once a password is set, you can add passkeys (Face ID, Touch ID, a security key) for quicker sign-in. Add them at **Settings → AX Code Desktop → Passkeys**.

Passkeys are tied to the current password. If you change or remove the password, saved passkeys are cleared and you'll add them again.

## Network boundary

- The Desktop web server and managed AX Code server accept only `localhost`, `127.0.0.0/8`, or `::1`.
- Proxy forwarding headers are ignored and persisted public origins are not trusted.
- SDK HTTP clients and server helpers reject remote AX Code URLs and network bind addresses.
- Git remotes and hosted model providers remain available; the restriction applies to access to the AX Code service itself.

Do not work around this boundary with an external proxy or tunnel. Use AX Code directly on the workstation that owns the code.
