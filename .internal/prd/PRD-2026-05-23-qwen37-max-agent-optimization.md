## Executive Summary

Qwen3.7-Max is a proprietary cloud model positioned by Alibaba/Qwen for agent-centric workloads: coding, office automation, MCP/tool workflows, long-context reasoning, and long-horizon autonomous execution. `ax-code` should leverage it as a premium cloud backend, but the product moat should remain provider-neutral agent orchestration: context packing, prompt-cache discipline, tool-loop reliability, and verification feedback.

This PRD defines the implementation path for making Qwen3.7-Max useful inside `ax-code` without turning the runtime into a Qwen-specific harness.

## Product Thesis

Qwen3.7-Max is most relevant to `ax-code` when the user is asking the model to act like a sustained engineering agent rather than a single-turn code generator. The product should optimize for the triangle the transcript highlights:

- **Capability:** can the model plan, inspect, patch, test, and recover across many files and tools?
- **Cost:** can the harness keep high-value stable context cached or summarized so long sessions remain affordable?
- **Stability:** can the loop avoid drift, repeated failed edits, stale assumptions, and false completion during long autonomous runs?

The optimized `ax-code` path should therefore be "premium long-agent execution with measured verification", not "always use the biggest Qwen context window".

The implementation unit is an **Agent Optimization Layer** around Qwen3.7-Max:

```text
task classifier / model router
→ repo context packer
→ planner and task decomposition
→ tool-call policy
→ verifier / test runner
→ patch safety and rollback evidence
→ long-task memory and failure traces
→ cost and stability telemetry
```

This layer is designed for Qwen3.7-Max first, but it must remain reusable across other premium cloud models and future local/open-weight helpers.

- Alibaba OpenAI-compatible reasoning models already receive `enable_thinking: true` and a clamped `thinking_budget` through `packages/ax-code/src/provider/transform.ts`.

- `.ax-code/memory.json` project memory exists and can inject path-scoped cached context into system prompt construction.
- Session token accounting already records cache read/write token fields when providers report them.
- System prompts already nudge agents toward `verify_project` and debug/impact tools where available.
- Quality rollout tooling already has benchmark, rollback, promotion, and reentry concepts that can inform the Qwen3.7-Max evaluation harness.

Gaps:

- `preserve_thinking` is not currently modeled as a provider option.
- OpenRouter's curated allowlist does not include `qwen/qwen3.7-max`.
- Prompt-cache breakpoints are not modeled as a provider-neutral request policy.
- Long-context repo packing is not yet an explicit mode/profile; context injection still relies on the existing memory, prompt, and compaction paths.
- There is no Qwen3.7-Max readiness probe or repo-local benchmark suite.
- Model routing is not yet cost/risk-aware; simple tasks and high-risk tasks do not have different policy paths.
- Patch scoring/rollback evidence exists in scattered runtime/quality surfaces, but there is no Qwen3.7-Max long-agent contract that requires verification before closure.

## Problem Statement

Qwen3.7-Max's value is unlocked by stable long context, reliable tool use, thinking continuity, and cache reuse. A naive integration that only exposes the model id will be expensive and fragile:

- Re-sending stable repo context every turn wastes tokens and latency.
- Enabling thinking without preserving useful reasoning state can lose long-horizon execution benefits.
- Dumping a large repository into 1M context can still degrade task focus.
- Provider-specific cache behavior may silently fail or increase cost if breakpoints are applied to dynamic content.
- Vendor benchmarks do not predict `ax-code` behavior on dirty worktrees, bug queues, provider changes, TUI/runtime failures, and narrow commits.
- Single-model lock-in ignores the practical lesson from the transcript: high agent capability is only useful when cost and stability remain inside operating bounds.

## Goals

- Make Qwen3.7-Max available through the right provider routes without breaking curated provider contracts.
- Add an explicit long-agent optimization profile that remains provider-neutral but has Qwen3.7-Max-specific defaults where documented.
- Represent thinking continuity and prompt caching as capability-gated policies.
- Improve context packing for 1M-context cloud backends without making "dump everything" the default.
- Add local evaluation so Qwen3.7-Max is promoted based on measured `ax-code` outcomes.
- Track Qwen3.7-Max quality as cost-per-verified-completion, not only pass/fail accuracy.
- Produce a coding-ready first slice that improves Qwen3.7-Max behavior without needing live provider credentials.

## Non-Goals

- Do not optimize local inference for Qwen3.7-Max; there are no open weights for this model.
- Do not make `ax-code` a Qwen-only harness.
- Do not enable OpenRouter explicit caching for Qwen3.7-Max until the route is probed.
- Do not raise Alibaba output/thinking caps globally as part of this PRD.
- Do not replace existing compaction or memory systems with a new parallel implementation.
- Do not encode unverified Qwen3/Qwen3.5 architecture or training claims as product assumptions for Qwen3.7-Max.

## User Stories

- As a maintainer, I can select Qwen3.7-Max as a premium model and get documented thinking/tool/cache behavior instead of a raw model id.
- As a user running a long repo task, I can use a long-agent profile that spends tokens on relevant repo context and verification, not repeated static prompt payload.
- As a maintainer evaluating model quality, I can compare Qwen3.7-Max against other providers on repeatable `ax-code` tasks.
- As an operator, I can see whether cache reads/writes and thinking tokens are helping or increasing cost.
- As a maintainer, I can decide when Qwen3.7-Max is worth using over cheaper/local alternatives based on capability, cost, and stability telemetry.
- As a future maintainer, I can train or tune smaller local planner/navigation models from `ax-code` traces without coupling that data pipeline to Qwen3.7-Max itself.
- As a user with Qwen3.7-Max selected, I can see a `Super-Long` toggle next to `Autonomous`, enabled by default for that model, so I can intentionally run a supervised long task with visible guardrails.

## Implementation Principles

- **No model-body tuning for Qwen3.7-Max.** It is a closed API model. Optimization happens in the agent layer.
- **Provider-neutral core, Qwen-specific edge.** Qwen defaults live in provider/profile policy, not scattered session logic.
- **Verifier before confidence.** Long-agent success must be backed by tests/checks or explicit verification envelopes.
- **Rollback evidence before broader autonomy.** High-risk edits need enough patch and failure evidence to reverse or diagnose.
- **Traces are assets.** Successful patches, failed patches, tool traces, test outputs, and cost data should become structured eval/training inputs.
- **Cross-check only where justified.** Claude/GPT cross-checking is reserved for high-risk refactors or release-critical changes because it can erase Qwen's cost advantage.
- **Super-Long is supervised, not unbounded.** The UI toggle enables long-run guardrails; it does not mean the model can run without budgets, checkpoints, verification, approval gates, and stop conditions.

Product behavior:

- Add a visible `Super-Long` button immediately to the left of the existing `Autonomous` button.
- Default state:
  - Off for all models by default.
  - On by default when the active model is Qwen3.7-Max.
  - User toggles always win over model defaults for the current session.
- The toggle enables supervised long-run behavior only when Autonomous mode is also enabled or when the runtime can enforce equivalent autonomous-loop guardrails.
- The hard maximum ceiling is 72 hours. No config or model default may exceed 72h.
- Super-Long must use paced execution by default. It should prefer steady progress over short request bursts because cloud providers can impose short-window rate limits that would otherwise block the whole run.
- Recommended presets:
  - `focus`: 30-60 minutes,
  - `deepWork`: 4-8 hours,
  - `research`: 24-72 hours, experimental and approval-gated.

Required guardrails:

- Goal contract: capture goal, scope, allowed files/tools, and explicit verifier commands before long-run starts.
- Time budget: bounded by preset/user config and capped at 72h.
- Cost budget: stop when estimated token/dollar budget is exceeded.
- Provider pacing budget: enforce per-provider request, token, tool-call, and concurrent-agent ceilings for short windows and rolling windows.
- Burst control: avoid launching many sub-agents or expensive model calls at once; queue work with jittered delays when provider limits are unknown.
- Backoff and recovery: on rate-limit or overload responses, pause, checkpoint, summarize current state, and resume only after exponential backoff or user approval when the delay exceeds the configured threshold.
- Checkpoint: write progress, diff summary, trace summary, and verification state at least every 30-60 minutes.
- Verifier: run test/lint/build or configured verifier after meaningful patch sets.
- Rollback evidence: preserve enough snapshot/patch metadata to revert or diagnose failed patch sets.
- Human approval gate: require approval before broad refactors, destructive file operations, cross-package rewrites, or external network/dependency changes.
- Stall detection: stop or ask for approval after repeated failures without new evidence.
- Progress report: produce reviewable status summaries at each checkpoint.
- Stop semantics: distinguish `complete`, `blocked`, `budget_exceeded`, `stalled`, `unsafe`, `approval_required`, and `user_stopped`.

Acceptance criteria:

- TUI shows `Super-Long` immediately left of `Autonomous`.
- Default is off for non-Qwen3.7-Max models and on for Qwen3.7-Max.
- Manual user toggle overrides model default.
- Runtime rejects any Super-Long ceiling above 72 hours.
- Runtime rejects or queues bursty execution plans that exceed configured provider pacing budgets.
- Rate-limit responses do not spin the agent loop; they produce checkpointed backoff state and visible progress/status.
- Tests prove Super-Long cannot silently bypass verifier, budget, checkpoint, or approval requirements.
- Tests cover short-window request throttling, concurrent-agent limits, backoff, and resume after provider pacing delays.
- The mode records trace/checkpoint artifacts suitable for Phase 6 evaluation reports.

## Metrics

- Task success rate on local eval tasks.
- Verified completion rate: completed tasks with passing tests/checks.
- Tool retry rate and repeated-failure rate.
- Cache read/write token ratio.
- Net input tokens per successful task.
- Wall-clock duration per successful task.
- Human intervention count.
- Cost estimate per successful task.
- Cost per verified completion.
- Provider pacing incidents: throttled requests, backoff duration, queued sub-agent work, and rate-limit recoveries.
- Stability score: no repeated failure loop, no false completion, no destructive unrelated edits.

## Risks and Mitigations

- **Provider behavior changes.** Keep provider-specific features gated by capability and route probes.
- **Cache support ambiguity.** Treat OpenRouter Qwen3.7-Max cache as unverified until a live route confirms it.
- **Cost blow-up from 1M context.** Use tiered context packing and cache only stable blocks.
- **Thinking-token cost increase.** Keep thinking budget clamped and disable thinking for auxiliary requests.
- **Qwen-specific overfitting.** Keep APIs provider-neutral and use model-specific defaults only at the edge.
- **Unverified architecture claims.** Treat transcript claims about Qwen3 data, MoE design, QK-Norm, and training stages as background only until a Qwen3.7-Max technical report is published.
- **False economy.** Lower token price does not guarantee lower total task cost if the model is verbose or repeatedly loops. Track cost per verified completion.
- **Dirty worktree churn.** Stage only PRD/ADR files and indexes if this PRD is committed later; `.internal` is gitignored by default.
